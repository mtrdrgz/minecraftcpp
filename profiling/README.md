# Performance Profiling Tools

This directory contains tools for profiling the Minecraft worldgen performance in the C++ port.

## Quick Start

### 1. Build and Run with Profiling Enabled

The profiler is automatically enabled. Just run the game normally:

```powershell
cd mcpp
cmake --build build --config Release
cd ..
.\mcpp\build\mcpp.exe --quickPlaySingleplayer
```

### 2. Generate Profiling Data

Simply play the game and let it generate chunks as normal. The profiling data is collected automatically while chunks are being generated. Exit the game to flush the profiling data to a CSV file.

**Profiling data is saved to:** `profiling/profiles/profile_YYYYMMDD_HHMMSS.csv`

### 3. Analyze the Data

```bash
# Analyze the latest profile
python profiling/tools/analyze_profile.py profiling/profiles/profile_YYYYMMDD_HHMMSS.csv

# Generate visualizations (requires matplotlib)
pip install matplotlib
python profiling/tools/visualize_profile.py profiling/profiles/profile_YYYYMMDD_HHMMSS.csv
```

Visualizations are saved to: `profiling/profiles/viz/`

## What Is Being Measured

The profiler instruments the following critical paths:

### Chunk Generation (Async - Worker Threads)
- **`fillFromNoise`** - Main terrain density sampling and interpolation
- **`buildSurface`** - Surface rule application (grass, sand, stone layering)
- **`applyCarvers`** - Cave and canyon carving

### Chunk Decoration (Main Thread)
- **`decorateChunk`** - Feature placement (trees, flowers, ores)
- **`runStructures`** - Structure piece assembly
- **`remesh_9chunk_neighborhood`** - Mesh updating after cross-chunk writes

### Startup
- **`startup_terrain_generation`** - Initial 5×5 chunk terrain gen (blocking)
- **`startup_decoration_and_structures`** - Initial decoration (blocking)

### Per-Thread Statistics
- Shows which worker threads are active and how balanced the load is
- Detects thread stalls or load imbalance

## CSV Output Format

```
Timestamp(s),Event,ChunkKey,Duration(ms),ThreadID
0.000,startup_terrain_generation,,2.345,<main>
0.002,startup_fillFromNoise,0,0,1.234,<main>
...
5.678,fillFromNoise,10,15,45.678,<worker-1>
5.679,buildSurface,10,15,23.456,<worker-1>
...
```

Fields:
- **Timestamp(s)**: Wall clock time since profiling started
- **Event**: Event name (what was being timed)
- **ChunkKey**: Chunk coordinates "cx,cz" or empty if N/A
- **Duration(ms)**: How long the event took
- **ThreadID**: Which thread executed it

## Key Metrics to Watch

### Performance Indicators

1. **`decorateChunk` duration**
   - If > 500ms avg: decoration is the bottleneck
   - Goal: < 100ms per chunk

2. **`fillFromNoise` duration**
   - If > 100ms avg: noise sampling is slow
   - Goal: < 50ms per chunk

3. **Thread utilization**
   - Check if all worker threads are active
   - If one thread dominates: load imbalance detected

4. **Startup times**
   - `startup_terrain_generation` should be < 1 second
   - `startup_decoration_and_structures` should be < 2 seconds

### Bottleneck Detection

Look for the event with the highest **Total(ms)** value. That's your bottleneck.

**Example analysis:**
```
If decorateChunk total >> fillFromNoise total:
  -> Decoration is the bottleneck (trees, features)
  -> Solutions: parallelize decoration, reduce feature count

If fillFromNoise total >> decorateChunk total:
  -> Terrain sampling is the bottleneck
  -> Solutions: optimize noise functions, cache values, reduce sampling
```

## Scripting & Automation

You can analyze multiple profiles:

```bash
# Analyze all profiles in a directory
for file in profiling/profiles/profile_*.csv; do
    echo "=== $file ==="
    python profiling/tools/analyze_profile.py "$file"
done
```

## Troubleshooting

**Q: No profiling data file generated**
- Ensure the game exited normally (didn't crash)
- Check that `profiling/` directory exists and is writable
- Check console output for error messages

**Q: Profile shows mostly zeros**
- Events may be running faster than timer resolution
- Increase scope to include multiple iterations

**Q: Thread IDs look strange**
- Thread IDs are OS-specific hexadecimal values
- Just use them to identify different threads

## Performance Optimization Workflow

1. **Run profiler** and collect baseline data
2. **Analyze** to identify bottleneck
3. **Hypothesize** what's causing it (noise function? RNG? feature placement?)
4. **Implement** targeted optimization
5. **Re-profile** to confirm improvement
6. **Repeat** for next bottleneck

## Integration with CI/CD

For automated regression testing, capture baseline profiles and compare against them:

```bash
# Baseline
python profiling/tools/analyze_profile.py baseline.csv > baseline_stats.txt

# After changes
python profiling/tools/analyze_profile.py new.csv > new_stats.txt

# Compare
diff baseline_stats.txt new_stats.txt
```

If new times are > 10% slower, fail the test.
